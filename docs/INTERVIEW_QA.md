基于 InternHub 项目整理的常见面试问题与参考回答，便于准备实习/校招面试。请结合个人实际开发经历调整表述。

## 一、项目与动机

**Q1：请简单介绍一下这个项目。**

InternHub 是一个实习/校招信息聚合与投递管理平台。用户可以看到职位列表、对感兴趣的职位投递、查看自己的投递记录和状态，并且有一个基于 AI 的岗位推荐功能，会根据用户未投递的职位给出排序和推荐理由。后端采用 Go 微服务架构，包含网关、认证、用户、职位、投递、推荐六个服务，前端用 React + TypeScript + Vite 实现，通过网关统一访问后端 API。

**Q2：为什么做这个项目？/ 项目解决了什么问题？**

希望把常见的“找实习/校招”流程做成一个可用的 Demo：集中展示职位、记录投递状态、避免重复投递，并且能结合 AI 做个性化推荐，减轻筛选成本。同时用微服务拆分，练习服务划分、网关、JWT、服务间 HTTP 调用以及前后端分离的完整链路。

**Q3：你在项目中负责哪些部分？**

（按真实分工回答）例如：我负责了推荐服务的整体设计与实现（从拉取职位和投递数据到对接 OpenAI、返回带理由的推荐列表）、推荐服务的单测、网关侧对推荐接口的代理与超时配置、前端从零搭建（React + Vite）、职位/投递/推荐/资料/登录注册等页面与接口对接，以及一键启动脚本和功能测试脚本的编写。

## 二、架构与设计

**Q4：为什么用微服务而不是单体？**

一方面是为了学习微服务拆分和网关模式，另一方面不同职责清晰分离：认证、用户资料、职位、投递、推荐各自独立部署和扩展。例如推荐服务只依赖 HTTP 调用其他服务，没有自己的数据库，将来若要换推荐算法或加缓存，只需改推荐服务。网关统一鉴权、路由和超时，前端只对接一个入口。

**Q5：认证与鉴权是怎么做的？**

认证由 auth-service 负责：注册、登录成功后签发 JWT，前端把 token 存 localStorage，请求时在 Header 里带 `Authorization: Bearer <token>`。网关用 JWT 中间件校验 token，解析出 `user_id`，在转发到 user、apply、recommend 等需要身份的接口时加上 `X-User-Id`，下游服务只信任网关注入的该 Header，不再解析 JWT。这样 JWT 密钥只存在于网关和 auth-service，下游无状态、易扩展。

**Q6：用户信息和用户资料为什么拆成两个服务？**

auth-service 只做“账号与认证”：邮箱、密码、登录态，是唯一用户数据源。user-service 只做“扩展资料”：昵称、头像等可选的展示信息。这样认证和资料可以独立演进（例如以后资料要加更多字段或迁到别的存储），且职责单一，符合微服务按边界拆分的原则。

**Q7：推荐服务没有自己的数据库，数据从哪来？**

推荐服务通过 HTTP 调用 job-service 拿全量职位列表、调用 apply-service 拿当前用户的投递列表、可选调用 user-service 拿昵称等做 AI 上下文。在内存里过滤掉已投递的职位得到“候选列表”，再根据是否配置 OpenAI API Key 决定是调用 AI 排序并生成理由，还是直接按时间倒序返回。这样推荐逻辑与职位、投递数据解耦，便于以后换数据源或加缓存。

## 三、技术实现

**Q8：推荐服务是如何对接 OpenAI 的？**

推荐服务里有一个 ai 包，用 HTTP 调用 OpenAI 兼容的 `/chat/completions` 接口。请求里把候选职位（id、title、company、link）、用户已投递的 job_id 列表、用户昵称拼成 JSON 作为 user 消息，system 里说明要返回 JSON 格式的 job_ids、reasons、summary。收到响应后解析 content（会去掉 markdown 代码块和末尾多余反引号），再按 job_ids 顺序和 reasons 组装成返回给前端的推荐列表。若未配置 API Key 或调用失败，会降级为按时间排序的未投递列表，并在返回里标记 `ai_used: false`。

**Q9：网关转发推荐请求为什么要把超时设成 60 秒？**

因为推荐接口内部会调 OpenAI，接口响应时间可能到十几秒甚至更长。原来网关默认 5 秒超时，经常还没等推荐服务返回就超时，前端收到“recommend service unavailable”。把转发推荐服务的 HTTP 客户端超时改成 60 秒后，正常情况都能拿到推荐结果。

**Q10：前端如何知道用户是否登录？**

前端在 api 层从 localStorage 读 token，有 token 就在请求头里带 `Authorization: Bearer <token>`。需要登录的页面（我的投递、推荐、资料）在加载时若发现没有 token 就重定向到登录页。登录成功后把返回的 access_token 写入 localStorage 并跳转到职位列表。退出时清除 token 并刷新页面。

**Q11：前端如何调用后端？有没有跨域问题？**

开发环境用 Vite 的 proxy 把 `/api` 和 `/health` 代理到 `http://localhost:8080`，浏览器只和 Vite 同源，没有跨域。生产环境需要把前端部署到同一域名下或由后端/网关配置 CORS。

## 四、测试与工程化

**Q12：推荐服务做了哪些测试？**

为 recommend-service 写了单元测试：ai 包测空 Key、模拟 HTTP 返回合法/非法 JSON、空 choices、非 200 等；handler 通过注入 Recommender 接口做 mock，测缺 X-User-Id、非法 user_id、服务报错、正常返回；service 里用 httptest 模拟 job/apply/user 三个上游，测“排除已投递”“全已投递”“空列表”以及 buildFallbackResult、buildResultFromAI 的组装逻辑。运行 `go test ./recommend-service/...` 可跑全量测试。

**Q13：如何做功能/联调测试？**

项目里有 `scripts/test-example.sh`：用新邮箱注册、登录拿 token、创建几条职位、拉推荐、投递两个职位、再拉推荐看条数是否少 2、更新昵称和查资料。全程用 curl 经网关调 API，适合验证前后端与各服务联调是否正常。启动用 `scripts/start-all.sh` 一键起六个服务，再跑测试脚本即可。

## 五、难点与收获

**Q14：开发过程中遇到的最大难点是什么？**

（示例）一个是推荐接口在网关侧 5 秒超时，导致前端经常拿不到推荐结果，后来查日志发现是“context deadline exceeded”，才把推荐这条代理单独改成 60 秒超时。另一个是 OpenAI 返回的 JSON 有时带 markdown 代码块或末尾多一个反引号，导致解析失败、走到降级逻辑，后来在解析前做了 trim 和去除反引号的处理，并加了 `ai_used` 和 summary 提示，方便排查是否真的用了 AI。

**Q15：如果继续做这个项目，你会做哪些改进？**

（可结合 README 或自己想法）例如：给职位加类型、城市、截止时间、分页和筛选；推荐结果加 Redis 缓存减少对 OpenAI 的调用；网关加限流和更严格的 CORS；各服务 Docker 化后用 docker-compose 一键起全量；用 GitHub Actions 跑测试和构建；前端加错误边界和加载骨架屏；为 auth、job、apply 等补更多单测和 E2E。

**Q16：这个项目让你对微服务/Go/React 有什么新的理解？**

（按个人体会答）例如：微服务拆得好可以做到“推荐服务无库、只调别人”，边界清晰；Go 的 httptest 和接口注入让单测写起来比较顺；网关统一鉴权和超时能避免每个服务都重复实现；前端通过一层 api 封装和 token 管理，页面只关心状态和 UI，接口细节集中在一处好维护。

## 六、简短话术（1 分钟版）

**用一分钟介绍 InternHub：**

InternHub 是一个实习校招的投递管理平台，后端用 Go 写了六个微服务：网关、认证、用户资料、职位、投递和推荐。用户注册登录后可以看职位、投递、查自己的投递记录，还有一个推荐接口会排除已投递的职位，并可以接 OpenAI 做排序和推荐理由。推荐服务没有自己的库，通过 HTTP 调职位和投递服务拿数据，再决定是调 AI 还是按时间排序。前端用 React 和 Vite 做了职位列表、投递、推荐、资料和登录注册，通过网关统一访问后端。我负责了推荐服务的设计与实现、前端搭建和联调、以及一键启动和测试脚本，过程中解决了网关超时和 AI 返回解析等问题。
